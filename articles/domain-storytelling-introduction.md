# 記事の概要
Domain Storytelling（ドメインストーリーテリング）について、その概念、手法、メリットを体系的にまとめる記事です。

# アウトライン案

## 1. はじめに
- Domain Storytellingとは何か？
  - 顧客や専門家（ドメインエキスパート）が語る「物語」を、ピクトグラムを使って可視化するモデリング手法。
- なぜ注目されているのか？
  - 複雑な業務フローを、技術者以外も理解できる共通言語で記述できる。
  - DDD（ドメイン駆動設計）の戦略的設計フェーズで有効。

## 2. Domain Storytellingの基本要素（The Pictographic Language）
- **Actors（アクター）**: 人、グループ、またはシステム（例：顧客、店員、決済システム）。
- **Work Objects（ワークオブジェクト）**: アクターが操作するもの（例：注文票、商品、メール）。
- **Activities（アクティビティ）**: アクターがワークオブジェクトに対して行う行動（例：作成する、渡す、承認する）。
- **Sequence Numbers（順序）**: 物語の時系列を示す番号。

## 3. ストーリーの語り方（モデリングの進め方）
- ワークショップ形式での実施方法。
- **Listen & Draw**: モデレーターが聞きながらリアルタイムに描くことの重要性。
- 具体的なステップ:
  1. アクターの特定
  2. ワークオブジェクトの特定
  3. アクティビティの記述
  4. 矢印でつないでストーリーを作る

## 4. 他の手法との比較
- Event Stormingとの違い
  - Event Storming: イベント（出来事）中心、大規模な全体像の把握に向く。
  - Domain Storytelling: ストーリー（会話・流れ）中心、具体的な業務フローの理解に向く。
- User Story Mappingとの違い

# 執筆に向けたリサーチ・アクション指示書

記事を執筆するにあたり、以下の点について調査・実践を行うことを推奨します。

## 1. 基礎知識のインプット
- **公式サイト**: [Domain Storytelling](https://domainstorytelling.org/) を一通り確認してください。特に "Quick Start" ガイドが参考になります。
- **書籍**: もし可能であれば『Domain Storytelling: A Collaborative Modeling Method』（Stefan Hofer, Henning Schwentner著）の序章や事例部分だけでも目を通すと深みが出ます。

## 2. 具体的に調べるべきポイント
- **「As-Is」と「To-Be」の使い分け**: 現状（As-Is）を理解するために使う場合と、あるべき姿（To-Be）を設計する場合でどう視点が変わるか。
- **アンチパターン**: 「細かすぎるストーリー」や「条件分岐の書きすぎ（基本はハッピーパスを描く）」など、初心者が陥りやすい罠について。

## 3. 実践アクション（やってみること）
実際に手を動かしてみるのが最も理解が早いです。以下の手順で小さなモデリングを行ってみてください。

1. **ツールの準備**: ブラウザで使える [WPS Domain Storytelling Modeler](https://www.wps.de/modeler/) を開いてください。
2. **身近な業務のモデリング**: 
   - 例：「プルリクエストのレビューからマージまでのフロー」や「毎朝のコーヒーを買うフロー」など、自分がよく知っているプロセスを一つ選んでください。
3. **描画**:
   - 登場人物（自分、同僚、GitHub Botなど）を配置。
   - やり取りするもの（コード、コメント、承認スタンプ）を配置。
   - 矢印でつなぎ、番号を振る。
4. **振り返り**:
   - 出来上がった図を見て、「この図だけで他人に説明できるか？」を確認してください。

## 4. 記事への反映
- 実践してみた感想（「意外とここが難しかった」「可視化すると無駄に気づけた」など）を記事に盛り込むと、オリジナリティが出ます。
